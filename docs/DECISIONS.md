# DECISIONS — технические решения

Журнал зафиксированных решений. Новые решения добавлять снизу с датой и обоснованием.
Формат записи: `## YYYY-MM-DD — Заголовок`, далее блоки **Context / Decision / Consequences**.

---

## 2026-05-23 — Язык: Python 3.11+

**Context.** Учебный проект, срок 1.5 дня. Нужен язык, на котором быстро поднимается Telegram-бот и есть зрелые библиотеки.
**Decision.** Python 3.11+.
**Consequences.** Большая экосистема, простые async-библиотеки. Минус — нужно следить за версией интерпретатора на деплое.

---

## 2026-05-23 — Фреймворк бота: aiogram 3.x

**Context.** Telegram Bot API, async, понятный роутинг, активно поддерживается.
**Decision.** aiogram 3.x.
**Consequences.** Современный API, удобные роутеры и фильтры. Не использовать примеры из aiogram 2 — они несовместимы.

---

## 2026-05-23 — Транспорт: Telegram Bot API (long polling)

**Context.** Демо для школьной защиты, временный деплой. Webhook требует HTTPS-домена и сложнее.
**Decision.** Long polling.
**Consequences.** Проще в запуске и отладке. На промышленный масштаб не годится, но для учебного — ок.

---

## 2026-05-23 — Хранение знаний: статичные файлы YAML/JSON

**Context.** Контент стабильный, обновлять его руками, БД пользователей не нужна.
**Decision.** Файлы в `data/` (предпочтительно YAML за читаемость; допустим JSON). Формат статьи: `id`, `title`, `body`, `disclaimer`, `sources`.
**Consequences.** Не нужна СУБД, миграции, ORM. Контент диффится в git.

---

## 2026-05-23 — Секреты: только в .env

**Context.** Токен бота и AI-ключи нельзя коммитить.
**Decision.** Все секреты — через переменные окружения, локально через `.env`. Шаблон — `.env.example`. `.env` — в `.gitignore`.
**Consequences.** На деплое переменные задаются через панель хостинга. Никаких хардкод-токенов.

---

## 2026-05-23 — AI-режим как отдельный изолированный слой

**Context.** AI должен помогать, но не быть источником истины и не ломать бот при отсутствии ключа.
**Decision.**
- AI-модуль — отдельный пакет `src/ai/`.
- Включается флагом `AI_ENABLED` + наличием API-ключа.
- Получает базу знаний как контекст.
- Системный промпт явно запрещает выдумывать юридические факты.
- Любая ошибка AI ловится и не валит остальной бот.

**Consequences.** Бот работает и без AI. На демо можно показать обе версии. Юридический риск ниже.

---

## 2026-05-24 — Docker: long polling, без webhook

**Context.** Stage 4 — деплой на VPS. Нужен простой способ запустить бота в контейнере.
**Decision.** `Dockerfile` + `docker-compose.yml`. Long polling (не webhook) — не нужен домен, HTTPS, nginx. `restart: unless-stopped` — бот поднимается автоматически после перезапуска сервера. Секреты через `env_file: .env`.
**Consequences.** Запуск одной командой: `docker compose up -d --build`. Webhook и Caddy/nginx — вне scope этого проекта.

---

## 2026-05-23 — Юридический контент: справочно + дисклеймер

**Context.** Школьники не должны брать на себя ответственность за юридическую консультацию.
**Decision.** Любой текст про законы/штрафы/зоны сопровождается дисклеймером и ссылкой на официальный источник. AI не имеет права давать конкретные правовые утверждения без опоры на базу знаний.
**Consequences.** Тексты получаются обтекаемее, но безопаснее. Часть точных норм заменяется на «уточняйте на …».

---

## 2026-05-24 — AI-клиент: библиотека `openai`

**Context.** Stage 3 требует вызова OpenAI-compatible API. Нужна одна библиотека, которая поддерживает `base_url` для сторонних провайдеров (Polza AI и др.).
**Decision.** Библиотека `openai>=1.0` — официальный клиент, поддерживает `AsyncOpenAI(base_url=..., api_key=...)` для любого совместимого провайдера. Импорт ленивый (внутри функции `ask()`): если библиотека не установлена, ошибка перехватывается и бот не падает.
**Consequences.** Одна зависимость покрывает всех OpenAI-compatible провайдеров. Переключение провайдера — только через `.env`.

---

## 2026-05-24 — Погодный API: Open-Meteo + aiohttp

**Context.** Stage 4 добавляет функцию «Погода для полёта». Нужен бесплатный погодный API без обязательного ключа.
**Decision.**
- Погода: [Open-Meteo](https://open-meteo.com) — бесплатный, без ключа, JSON, возвращает `temperature_2m`, `wind_speed_10m`, `precipitation`. URL: `https://api.open-meteo.com/v1/forecast`.
- Геокодинг: Open-Meteo Geocoding API — тот же провайдер, без ключа. URL: `https://geocoding-api.open-meteo.com/v1/search`.
- HTTP-клиент: `aiohttp>=3.9` — уже транзитивная зависимость aiogram, добавлена явно т.к. используется напрямую.
- Логика оценки: ветер ≥ 10 м/с или осадки > 0 → ⛔; ветер 5–9.9 м/с → ⚠️; иначе → ✅.
- Оценка справочная, дисклеймер обязателен в каждом ответе.

**Consequences.** Нет платных API. Нет новых секретов. Одна зависимость (`aiohttp`). Если Open-Meteo недоступен — бот отвечает сообщением об ошибке и не падает.

## 2026-05-24 — Обратная связь: файл logs/feedback.txt

**Context.** Stage 5 добавляет кнопку «Оставить вопрос» при fallback. Нужно куда-то сохранять вопросы пользователей без БД.
**Decision.** Текстовый файл `logs/feedback.txt`: одна строка на вопрос, формат `[ISO-timestamp] текст`. Папка `logs/` создаётся автоматически (`Path.mkdir(exist_ok=True)`). В файл пишется только текст и время — без user_id, username или chat_id, чтобы не хранить персональные данные. `logs/` добавлен в `.gitignore`.
**Consequences.** Нет новых зависимостей. Простой просмотр через `cat logs/feedback.txt`. Если бот перезапускается или переезжает на новый сервер — файл не переносится автоматически (для учебного проекта ок).

---

## 2026-05-24 — FSM для AI-режима: aiogram MemoryStorage

**Context.** Нужно отличать «обычный текст пользователя» от «вопроса к AI после нажатия кнопки».
**Decision.** Используем встроенный aiogram FSM с `MemoryStorage`. Состояние `AiState.waiting_question` устанавливается при нажатии «🤖 Спросить ИИ» и сбрасывается после ответа или при переходе в меню. `MemoryStorage` достаточно — состояние сбрасывается при перезапуске бота, это приемлемо для учебного проекта.
**Consequences.** Нет внешних зависимостей для хранения состояния. При перезапуске бота пользователь выходит из AI-режима автоматически.
